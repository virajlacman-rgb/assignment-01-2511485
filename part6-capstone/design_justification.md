## Storage Systems

I would use multiple storage systems because the hospital’s goals involve transactional processing, analytics, semantic search, and real-time streaming. For patient readmission prediction, I would use a Data Lake to store raw historical treatment data, discharge summaries, and other large medical records. This is suitable because machine learning requires flexible access to large volumes of raw and semi-structured data. For doctors to query patient history in plain English, I would use a Vector Database. Clinical notes, discharge summaries, and patient history documents can be converted into embeddings and stored as vectors, allowing semantic search instead of only keyword matching. For monthly management reporting such as bed occupancy and department-wise costs, I would use a Data Warehouse because it is optimized for structured historical analysis, aggregations, and dashboard reporting. For real-time ICU vitals, I would use a streaming pipeline connected to a Data Lake or time-series style storage area so that high-volume device data can be ingested continuously and stored efficiently. The OLTP hospital system would still remain the main source for current patient records, appointments, and treatment transactions.

## OLTP vs OLAP Boundary

The OLTP system ends where day-to-day transactional hospital operations end. This includes patient registration, appointment updates, treatment entries, prescriptions, and billing events. These activities require fast writes, high consistency, and reliable current-state data. The analytical system begins once transactional data is extracted from OLTP and moved into downstream platforms such as the Data Warehouse, Data Lake, and Vector Database. The Data Warehouse supports structured reporting and trend analysis, the Data Lake supports large-scale historical and machine learning workloads, and the Vector Database supports semantic retrieval over clinical text. In this design, OLTP handles operational hospital workflows, while OLAP and AI systems handle reporting, prediction, and search.

## Trade-offs

One significant trade-off in this design is architectural complexity. Using multiple systems improves performance and suitability for different workloads, but it also increases integration effort, governance complexity, and operational overhead. Data must be synchronized carefully between OLTP, the warehouse, the lake, and the vector database. To mitigate this, I would use strong data pipelines, clear ownership of each storage layer, consistent patient identifiers, and data quality validation checks. This reduces duplication problems and helps ensure that each system remains reliable and aligned with the hospital’s reporting and AI needs.
